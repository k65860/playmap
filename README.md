# Playmap
아이의 기질과 놀이 키워드를 기반으로 맞춤형 놀이활동을 추천하고 기록할 수 있는 서비스

<br />

## 프로젝트 소개

**Playmap**은 아이의 기질 분석 결과와 기록자가 선택한 놀이 키워드를 기반으로 맞춤형 놀이활동을 추천하는 서비스입니다.

기존 기업 서비스에 신규 기능을 추가하는 프로젝트로 참여했으며, 기업에서 전달한 요구사항을 분석한 뒤 놀이활동 기록 기능을 React 기반으로 구현했습니다.

사용자는 캘린더를 통해 날짜별 놀이활동 기록을 확인할 수 있고, 바텀시트 모달에서 키워드를 선택한 뒤 추천 활동을 확인하고 기록을 저장할 수 있습니다.

## 주요 기능

### 1. 놀이활동 기록 캘린더

사용자가 월별 캘린더에서 날짜별 놀이활동 기록 여부를 확인할 수 있도록 구현했습니다.

* `react-calendar` 라이브러리를 활용한 월별 캘린더 UI 구현
* 날짜별 활동 기록 유무 표시
* 선택한 날짜 기준 활동 기록 목록 렌더링
* 기록 제목 클릭 시 상세 조회 흐름 연결

<br />

### 2. 바텀시트 기반 기록 생성/조회 모달

모바일 서비스 흐름에 맞춰 바텀시트 형태의 모달 UI를 구현했습니다.

* 날짜, 시간, 키워드, 활동 내용 입력 처리
* 선택한 키워드 요약 표시
* 추천 활동 확인 후 기록 작성 흐름 연결
* 저장 버튼 클릭 시 입력 값과 선택 키워드를 생성 API로 전달

<br />

### 3. 키워드 기반 필터링 및 추천 활동 표시

놀이활동 기록 과정에서 사용자가 선택한 키워드를 기반으로 추천 활동을 확인할 수 있도록 구현했습니다.

* `누구와`, `어디에서`, `어떤 활동을`, `얼마나` 기준의 키워드 필터링 UI 구현
* 선택된 키워드를 요약해 보여주는 UI 구현
* 선택 키워드 기반 추천 활동 표시
* 추천 활동 확인 후 놀이활동 기록 작성 흐름 연결

<br />

### 4. 기업 요구사항 명세화

기업에서 전달받은 신규 기능 요구사항을 분석하고, 팀원이 구현 범위를 빠르게 파악할 수 있도록 기능 단위로 정리했습니다.

* 기존 서비스 구조 분석
* 페이지별 기능 정리
* 신규 개발이 필요한 영역 표시
* 프론트엔드 화면 개발과 백엔드 API 연동 기준 정리
* Notion 기반 기능 명세서 작성

<br />

## 담당 역할

### 요구사항 분석 및 기능 명세

기업에서 전달받은 신규 기능 요구사항이 페이지별로 흩어져 있어 프론트엔드와 백엔드가 구현 범위를 동일하게 이해하기 어려운 문제가 있었습니다.

이를 해결하기 위해 기존 서비스의 페이지 구조를 먼저 분석하고, 신규 개발이 필요한 기능을 페이지 단위로 정리했습니다. 이후 Notion 문서에 화면별 기능, 필요한 데이터, API 연동 범위를 정리하여 팀원들이 같은 기준으로 개발할 수 있도록 했습니다.

<br />

### Figma 프로토타입 제작

놀이활동 기록 기능의 사용자 흐름을 정리하기 위해 Figma로 화면 구조를 설계했습니다.

특히 캘린더에서 날짜를 선택한 뒤 기록을 생성하고, 키워드를 선택하고, 추천 활동을 확인한 후 저장하는 흐름이 하나의 화면 안에서 자연스럽게 이어지도록 바텀시트 모달 구조를 설계했습니다.

<br />

### 놀이활동 기록 페이지 구현

`react-calendar`를 활용해 월별 캘린더 UI를 구현하고, 기록이 존재하는 날짜에는 표시점을 렌더링했습니다.

사용자가 날짜를 선택하면 해당 날짜의 기록 목록을 확인할 수 있으며, 기록이 없는 날짜에서는 새 기록을 생성할 수 있도록 흐름을 구성했습니다.

<br />

### 키워드 선택 및 추천 활동 흐름 구현

바텀시트 모달 내부에서 키워드 선택, 선택 키워드 요약, 추천 활동 표시, 기록 입력, 저장 API 연동까지 하나의 사용자 흐름으로 연결했습니다.

이를 통해 사용자가 키워드를 선택한 뒤 추천 활동을 확인하고, 같은 화면 안에서 바로 놀이활동 기록을 작성할 수 있도록 구현했습니다.

<br />

## 문제 해결 경험

### 1. 캘린더에서 기록 여부를 직관적으로 확인하기 어려운 문제

#### 문제

사용자가 월별 캘린더에서 어떤 날짜에 놀이활동 기록이 있는지 바로 파악하기 어려웠습니다.

#### 해결

`react-calendar`의 날짜 타일 렌더링 방식을 활용해 기록이 존재하는 날짜에 표시점을 추가했습니다.

#### 결과

날짜를 선택하기 전에도 기록 여부를 확인할 수 있어 활동 기록 탐색 흐름이 더 직관적으로 개선되었습니다.

<br />

### 2. 기업 요구사항을 팀 개발 범위로 정리해야 했던 문제

#### 문제

기업에서 전달한 신규 기능 요구사항이 페이지별로 흩어져 있어 프론트엔드와 백엔드가 구현 범위를 동일하게 이해하기 어려웠습니다.

#### 해결

기존 서비스의 페이지 구조를 분석한 뒤, 페이지별 기능과 신규 개발이 필요한 항목을 Notion 명세서로 정리했습니다.

#### 결과

팀원들이 구현해야 할 범위를 빠르게 파악할 수 있었고, 프론트엔드 화면 개발과 백엔드 API 연동 기준을 맞추는 데 도움이 되었습니다.

## 배운 점

이번 프로젝트를 통해 단순히 화면을 구현하는 것뿐만 아니라, 기업 요구사항을 실제 개발 가능한 기능 단위로 정리하는 과정의 중요성을 경험했습니다.

특히 프론트엔드와 백엔드가 같은 기준으로 개발하기 위해서는 화면 흐름, 필요한 데이터, API 연동 기준이 명확하게 정리되어야 한다는 것을 배웠습니다.

또한 `react-calendar`를 활용해 날짜별 기록 여부를 표시하고, 바텀시트 모달 안에서 키워드 선택부터 추천 활동 확인, 기록 저장까지 하나의 사용자 흐름으로 연결하며 사용자 데이터와 UI를 자연스럽게 연결하는 경험을 쌓았습니다.
